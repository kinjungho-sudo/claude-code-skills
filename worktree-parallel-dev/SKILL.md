# Worktree 병렬 개발 (Parallel Worktree Dev)

## 이 스킬의 목적

사용자는 **비개발자(PM 역할)**다. 클로드 코드 인스턴스를 여러 개 띄워 **동시에(병렬)** 개발해 속도를 내고 싶어 한다.
하지만 git을 직접 치지 않고, 충돌·꼬임·실수도 두렵다.

이 스킬은 그 상황에서 **(1) 구조를 쉬운 말로 이해시키고 (2) 인스턴스별로 무엇을 시킬지 알려주고 (3) git이 꼬이지 않게
지켜야 할 규칙과, 꼬였을 때 복구법까지** 안내한다. 핵심 원칙: **사용자는 쉬운 한국어로 지시하고, git 명령은 클로드가 친다.**

---

## 1. 핵심 멘탈 모델 — "복사본 여러 벌"

git **worktree**는 같은 레포(저장소)를 **여러 폴더에 통째로 복사**해 두고, 각 복사본에서 **다른 브랜치**를 동시에 작업하게 해준다.

```
프로젝트/
├── repo/                ← 원본 (통합·배포 담당, 보통 dev 브랜치)
│   └── (앱 소스 전체)
├── repo-editor/         ┐  각 폴더는 레포를 "통째로" 복사한 것.
│   └── (앱 소스 전체)   │  → 안에 모든 파일이 다 들어있다.
├── repo-api/      (동일) │  → 도메인별로 "쪼갠" 게 아니라,
└── repo-landing/  (동일) ┘    "복사본마다 한 도메인만 만지기로 약속"한 것.
```

> ⚠️ 자주 하는 오해: "editor 폴더 = 에디터 파일만 있는 폴더"가 **아니다.** 레포 전체 복사본이라 모든 파일이 다 있다.
> 단지 **그 폴더(창)에서는 에디터만 건드리기로 약속**할 뿐이다. 그래서 같은 파일이 여러 폴더에 중복으로 보이는 건 정상이다.

**창(폴더) 하나 = 클로드 인스턴스 하나 = 담당 도메인 하나.** 필요한 도메인 폴더만 열면 된다(전부 띄울 필요 없음).

---

## 2. 3층 브랜치 + 2단계 승인 게이트

```
feat/*   ← 각 복사본에서 '쓰는' 곳 (feat/editor, feat/api, ...)
  │  ① 승인: "feat/○○ dev에 합쳐줘"   (개발 통합 게이트)
dev      ← '모으는' 통합 브랜치 (여기선 직접 코딩 안 함)
  │  ② 승인: "dev를 main에 배포해줘"   (운영 배포 게이트)
main     ← 운영 (배포되면 실제 서비스에 나감)
```

- **복사본에서 작업한 건 자동으로 원본에 반영되지 않는다.** 사용자가 **합치기를 승인**해야 dev로 들어온다.
- 승인 게이트가 **2번**: ① 복사본→dev(개발 통합) ② dev→main(운영 배포, 더 신중).
- "덮어쓰기"가 아니라 **"합치기(merge)"** 다 — 통째로 덮는 게 아니라 **바뀐 부분만** 골라 합친다.

---

## 3. 역할별로 무엇을 시키나

### A. feat 인스턴스 (복사본에서 작업하는 클로드)
그 도메인 폴더의 창에서 클로드에게 순서대로:

```
1) 이 폴더에서 처음이면: 의존성 한 번 설치해줘   (worktree는 node_modules 비공유 → 폴더마다 1회)
2) 너 지금 어느 브랜치인지 확인해줘             (feat/editor 식으로 나와야 정상)
3) 작업 전에 dev 최신으로 맞춰줘               (git rebase origin/dev)
4) [원하는 기능] 만들어줘
5) 다 됐으면 빌드 확인하고 커밋해줘            ← 커밋은 "시켜야" 함
6) dev 최신 한 번 더 맞춰서 푸시까지 해줘
```

### B. 총괄(PM) 인스턴스 (원본 repo, dev 브랜치)
다른 인스턴스들이 푸시하면 PM 창의 클로드에게:

```
"feat/editor랑 feat/api 가져와서 dev에 합쳐줘"     ← ① 통합 게이트
(클로드: 합치고 → 빌드/검증 → 결과 보고. 충돌나면 정리)
"문제없으면 dev를 main에 배포해줘"                 ← ② 배포 게이트
```

> **커밋·푸시·병합·배포는 사용자가 명시적으로 시킬 때만** 실행한다. 묻지 않고 먼저 푸시/배포하지 않는다.

## 4. 꼭 이해할 개념 — "dev 최신화를 안 하고 병렬 시작하면?"

(비개발자가 가장 불안해하는 부분. 결론: **대부분 괜찮고, 손실은 없으며, 복구된다.**)

git은 합칠 때 **"공통 조상"**을 찾아 **각자 바뀐 부분만** 비교해 합친다. 그래서 옛날 dev에서 시작했어도,
그동안 dev에 생긴 변화가 **사라지거나 덮어써지지 않는다.**

```
dev:        A → B → C        (C = 작업자가 모르던 새 dev 변화)
                 ↘
feat/editor:      B → E       (옛날 B에서 시작해 만든 작업 E)

합치면:   C(원본 변화)  +  E(내 작업)  →  둘 다 살아남음
```

### 그래서 진짜 위험은 2가지뿐

| 위험 | 성격 | 어떻게 되나 |
|---|---|---|
| **① 충돌(conflict)** | 시끄럽지만 **안전** | 놓친 변화와 내 작업이 **같은 파일의 같은 줄**을 건드렸을 때. git이 **멈춰서 "뭘 쓸지 정해줘"** 라고 알림. 데이터 손실 없음 → PM이 정리하면 끝. |
| **② 조용한 논리 깨짐** | **진짜 조심** | git은 **글자가 겹치는 것만** 감지. 예: dev가 함수 이름을 바꿨는데 내 작업은 다른 파일에서 옛 이름을 호출 → **충돌 없이 깨끗이 합쳐지지만 코드가 깨짐.** → **합친 뒤 빌드/테스트로 잡는다.** |

### 요약

- **작업·데이터 손실: 없음** (git이 공통 조상 기준으로 다 보존)
- **충돌 가능성: 올라감** (그래도 멈춰서 알려주고, 복구 가능)
- **조용한 논리 깨짐: 가능** → **합친 뒤 `build`/테스트 필수**
- **복구: 항상 가능**

### "push 전 dev로 rebase(최신화)" 규칙이 있는 진짜 이유
merge가 망가져서가 **아니다.** 충돌을 **나중에 PM 창에서 한꺼번에** 만나는 대신,
**각자 작업 중인 본인 창에서 미리(맥락이 살아있을 때)** 만나 처리하는 게 훨씬 쉽기 때문이다.

---

## 5. 절대 규칙 (충돌·꼬임 예방의 핵심)

| ❌ | ✅ |
|---|---|
| 한 폴더(창)에서 여러 도메인 파일을 만짐 | **폴더당 한 도메인만** (같은 파일 동시편집이 충돌의 주원인) |
| 옛날 상태로 그냥 push | **push 전 항상 `git rebase origin/dev`** (non-fast-forward 거부·충돌 예방) |
| 합치고 검증 없이 배포 | **합친 뒤 `build`/테스트** → 조용한 논리 깨짐 차단 |
| 한 폴더에서 dev 서버 2개 이상 | **폴더당 dev 서버 1개** (여러 개가 같은 빌드 캐시를 공유하면 캐시 오염으로 Server Error. 병렬은 반드시 worktree=폴더별 캐시) |
| 묻지도 않고 push/merge/배포 | **사용자가 시킬 때만** 커밋·푸시·병합·배포 |
| main에 직접 커밋 | dev에서 통합 후 **PM만** main 배포 |

## 6. worktree 셋업·정리 (부록 — 클로드가 칠 명령)

```bash
# 만들기 (도메인별 폴더 + feat 브랜치)
git worktree add ../repo-editor  -b feat/editor
git worktree add ../repo-api     -b feat/api

# 현황 보기
git worktree list

# 정리 (폴더 제거 — feat 브랜치 자체는 남음, 나중에 재생성 가능)
git worktree remove ../repo-editor

# 첫 사용 시 폴더마다 1회 (node_modules는 worktree끼리 공유 안 됨)
#   cd ../repo-editor && (해당 앱 폴더에서) 패키지 설치
```

- `.env*.local` 같은 gitignore된 로컬 설정은 worktree에 **자동 복사되지 않는다** → 필요하면 원본에서 복사.
- worktree가 너무 많아 헷갈리면 **자주 쓰는 것만 남기고 정리**해도 된다(브랜치는 보존되므로 안전).

## 7. 언제 worktree를 쓰고, 언제 안 쓰나

- **쓸 때:** 서로 무관한 작업을 **여러 클로드 인스턴스로 진짜 동시에** 진행할 때. (폴더·캐시가 분리돼 파일 충돌 원천 차단)
- **안 써도 될 때:** 혼자 순차로 작업하면 충분할 때. worktree는 디스크 여러 배 + 폴더마다 의존성 설치라 **비용이 있다.**
- **교차 관심사(예: 다국어처럼 전 영역을 가로지르는 한 작업):** 병렬로 쪼개지 말고 **한 인스턴스 + 서브에이전트**로 한 번에 처리(영역이 겹쳐 충돌 위험).

---

## 8. 한 줄 요약

> 복사본(worktree)마다 **한 도메인만**, **push 전 최신화**, **합친 뒤 빌드**, **커밋·배포는 시킬 때만.**
> 최신화를 깜빡해도 손실은 없다 — 충돌은 멈춰서 알려주고, 조용한 깨짐은 빌드가 잡는다.
